Building CortexMap: AI-Powered Brain Atlas from 10,000 Research Papers

Engineering Deep Dive · Penn State · February 2026
Rust PostgreSQL RAG LLMs Vector Search Distributed Systems

CortexMap processes thousands of neuroscience research papers to automatically generate comprehensive profiles for brain regions. For each region in a predefined list (hippocampus, amygdala, prefrontal cortex, etc.), the system synthesizes its functions, associated diseases, and affected body parts from the literature—complete with citations. It's a batch generation system using distributed task queues, vector databases, and retrieval-augmented generation to turn literature into structured knowledge.

The Problem: Information Overload in Neuroscience

Neuroscience researchers face an impossible task: staying current with the literature. PubMed Central indexes over 7 million biomedical papers, with thousands added weekly. For any given brain region—say, the hippocampus—there might be 15,000 relevant papers spanning decades of research.

No researcher can read 15,000 papers. Creating comprehensive profiles for each brain region—documenting functions, disease associations, and systemic effects—requires synthesizing findings across hundreds of papers through manual reading and integration. This process takes months and is outdated by completion.

We built CortexMap to automate this synthesis. Given a list of brain regions, the system ingests thousands of relevant papers, extracts findings using vector search and large language models, and generates structured profiles (summary, diseases, affected body parts) with citation tracking. Every claim links back to specific passages in source papers.

System Architecture: Three Services, One Goal

CortexMap is built as three independent Rust microservices communicating over HTTP. This separation emerged from the distinct computational profiles of each task: I/O-bound paper fetching, CPU/GPU-intensive NLP processing, and lightweight orchestration.

Service Responsibilities

Fetcher Service: Handles all interaction with PubMed Central. When given a search query like "hippocampus AND memory formation", it queries the NCBI E-utilities API, retrieves up to 20 paper IDs, and queues download tasks. Workers asynchronously fetch full-text XML, upload to S3, and mark tasks complete. The architecture supports dozens of concurrent workers claiming tasks from PostgreSQL without conflicts.

BrainAtlas Service: The NLP workhorse. It downloads papers from S3, chunks them into 1000-character segments with 200-character overlap, generates vector embeddings via OpenRouter's API, and stores them in PostgreSQL with pgvector. Then it runs a retrieval-augmented generation (RAG) loop: the LLM searches for relevant chunks, reads them, and synthesizes a final summary. Every generated claim tracks back to specific character ranges in source papers.

Orchestrator Service: The coordinator. It generates diverse PubMed queries using an LLM (e.g., "hippocampus anatomy", "hippocampal lesions memory deficits"), submits them to the Fetcher, tracks progress in batches, and triggers BrainAtlas processing when papers arrive. A background watcher polls every 30 seconds for completed batches, ensuring the pipeline never stalls. Redis caches responses to reduce database load.

Why separate services?

Fetching papers is I/O-bound—workers spend most of their time waiting on network calls. LLM processing is API-rate-limited and memory-intensive. Orchestration is lightweight HTTP routing. Running them in one monolith would waste resources. Splitting lets us scale the Fetcher horizontally with more workers while running a single BrainAtlas instance, optimizing for each workload's characteristics.

The Distributed Paper Fetcher

Academic papers don't download themselves, and PubMed Central's API has rate limits. Our fetcher implements a production-grade distributed task queue using PostgreSQL, with component-level retry tracking and graceful failure handling.

The Task Queue

When the Orchestrator submits a query, the Fetcher creates one database row per paper in the fetch_tasks table. Each task starts with status='pending' and awaits a worker. Workers claim tasks using PostgreSQL's FOR UPDATE SKIP LOCKED feature:

"When a worker executes SELECT ... FOR UPDATE SKIP LOCKED, the database locks the row atomically. If another worker already holds the lock, PostgreSQL skips that row instead of blocking. The second worker immediately gets the next available task. No distributed locks, no Redis coordination, no race conditions—the database handles concurrency natively."

This pattern is beautifully simple. We tested it with 10 concurrent workers hammering a queue of 500 tasks—zero duplicate claims, zero deadlocks, linear scaling.

Component-Level Retries

Papers have three components: metadata (title, authors, abstract), full-text XML, and PDFs. Each component can fail independently. Instead of retrying the entire paper when a PDF download times out, we track each component's state separately.

Here's a concrete example. A worker picks up a task for paper PMC12345:

  1. Metadata fetch: Success. ESummary API returns JSON. Uploaded to S3. Component marked completed.
  2. Full-text fetch: Success. Downloaded 47KB XML file. Uploaded to S3. Component marked completed.
  3. PDF fetch: Network timeout after 30 seconds. Attempt counter increments (1/5). Component reset to pending.

The worker releases the task back to the queue. Five minutes later, another worker (or the same one) claims it again. It sees metadata and full-text are already done, so it skips directly to the PDF. This time, the download succeeds. Task complete.

Impact: On a batch of 500 papers with a 15% PDF failure rate on first attempt, this approach saved approximately 1,350 redundant API calls and 23 hours of wasted download time compared to task-level retries.

Handling Worker Crashes

Workers can crash, servers can restart, networks can partition. If a worker dies mid-processing, that task would be locked forever. We prevent this with heartbeat-based leases.

When a worker claims a task, it sets heartbeat_at to the current timestamp and saves its worker_id. A recovery job scans for tasks where heartbeat_at is older than 60 seconds. Those tasks get reset: status back to pending, worker fields cleared. Another worker can claim them.

This is lease-based processing. If the worker is still alive but just slow, it'll try to mark the task complete and get a database conflict error—the row was already reassigned. The worker logs the error and moves on. No duplicate work, no stuck tasks.

Real-world resilience:

During development, we had a worker that would occasionally hang on PDF downloads (PubMed Central sometimes returns malformed content-length headers). The heartbeat system automatically recovered those tasks within 60 seconds. We added better error handling later, but the heartbeat mechanism prevented hours of debugging stalled pipelines.

Retrieval-Augmented Generation with Vector Search

Fetching papers is just the beginning. The real challenge is synthesis: how do you turn 50 papers about the hippocampus into a coherent summary that accurately represents the literature?

This is where retrieval-augmented generation (RAG) shines. Instead of asking an LLM to hallucinate facts about the hippocampus, we give it access to a search engine over the actual paper content. The LLM retrieves relevant passages, reads them, and cites them in its answer.

Step 1: Text Chunking and Embedding

Papers are long—a typical neuroscience article is 5,000-8,000 words. LLMs have context windows (GPT-4o supports 128K tokens, but feeding entire papers wastes tokens and increases latency). We chunk papers into overlapping segments:

  • Chunk size: 1,000 characters (~200 words)
  • Overlap: 200 characters between chunks
  • Metadata: Each chunk stores source_s3_key, source_pmc_id, char_start, char_end

Each chunk gets converted into a vector embedding—a 1,536-dimensional array of floats that captures semantic meaning. We use OpenRouter's embedding API (typically OpenAI's text-embedding-3-small). Chunks with similar meanings have similar vectors in embedding space.

These embeddings go into PostgreSQL with the pgvector extension. Pgvector adds a vector data type and indexes for fast similarity search using HNSW (Hierarchical Navigable Small World graphs). We can query "find the 10 chunks most similar to 'hippocampal role in spatial memory'" in milliseconds.

Step 2: The RAG Loop

Now comes the magic. We prompt the LLM with:

"You are a neuroscience expert writing a comprehensive summary of the hippocampus. You have access to a search tool that retrieves relevant passages from research papers. Use this tool to gather evidence before writing your summary. Cite specific passages using [chunk:id] markers."

The LLM doesn't have knowledge of hippocampus research—it has a tool: search_embeddings(query: str, top_k: int). Here's how it uses it:

  1. Round 1: LLM calls search_embeddings("hippocampus anatomy structure", 10). We convert "hippocampus anatomy structure" to a vector, search pgvector, return 10 closest chunks.
  2. Round 2: LLM reads those chunks, then calls search_embeddings("hippocampal formation memory consolidation", 10). Gets 10 more chunks.
  3. Round 3: LLM calls search_embeddings("CA1 CA3 subregions electrophysiology", 10).
  4. Round 4: LLM decides it has enough information. Returns final summary with citations like "The hippocampus plays a critical role in memory consolidation [chunk:a3f7]..."

We limit the loop to 5 rounds to prevent runaway costs. In practice, most summaries complete in 3-4 rounds. The LLM is remarkably good at asking targeted questions—it starts broad ("hippocampus function") and drills down ("dentate gyrus neurogenesis").

Step 3: Citation Resolution

When the frontend renders a summary, it sees markers like [chunk:a3f7]. It queries the backend: "Give me metadata for chunk a3f7." The backend joins the brain_region_embeddings table with papers and returns:

{ "chunk_id": "a3f7", "source_pmc_id": "PMC3847291", "source_query": "hippocampus AND spatial memory", "char_start": 12847, "char_end": 13847, "text": "...studies have shown that hippocampal neurons..." }

The frontend renders this as a clickable citation bubble. Users can hover for a preview or click to open the paper on PubMed Central, jumping directly to the cited section.

Why this matters:

Without citations, AI-generated summaries are unverifiable. With our approach, every claim is traceable. If a user reads "The hippocampus contains place cells that encode spatial location" and wonders about the evidence, they can click the citation and land on the 1971 O'Keefe & Dostrovsky paper that discovered place cells. Trust through transparency.

Orchestrating Async Workflows

The Fetcher and BrainAtlas services are async-first. They don't wait for long-running operations—they queue work and return immediately. The Orchestrator's job is to coordinate these async workflows without blocking.

Batch-Based Processing

When a user requests a summary for the hippocampus, here's what happens behind the scenes:

  1. Query Generation: Orchestrator calls BrainAtlas to generate 3-5 diverse PubMed queries using an LLM. Example queries: "hippocampus anatomy", "hippocampal formation memory consolidation", "CA1 CA3 subfields electrophysiology".
  2. Batch Creation: Orchestrator creates a region_processing_batch row with status='collecting'. It stores the generated queries and sets expected_task_count (typically 60-100 papers across all queries).
  3. Fetcher Invocation: For each query, Orchestrator calls POST /fetcher-be/api/queue/enqueue. Fetcher returns task IDs immediately (e.g., [task_1, task_2, ..., task_20] for a query that found 20 papers).
  4. Batch Tracking: Orchestrator stores all task IDs in batch.fetch_task_ids[] and returns the batch_id to the frontend. The user sees "Fetching papers... 0/83 complete".

At this point, the Orchestrator is done. It doesn't poll, doesn't wait, doesn't block. Workers in the Fetcher service pick up those 83 tasks over the next few minutes. The Orchestrator has moved on to serve other requests.

The Completion Watcher

So how does the Orchestrator know when fetching finishes? A background task called the completion watcher runs in a loop:

loop { sleep(30 seconds) batches = query("SELECT * FROM region_processing_batches WHERE status IN ('collecting', 'ready')") for batch in batches { task_statuses = query("SELECT status FROM fetch_tasks WHERE id IN $1", batch.fetch_task_ids) if all_tasks_completed(task_statuses) { update batch.status = 'ready' invalidate_cache(batch.id) } } ready_batches = query("SELECT * FROM region_processing_batches WHERE status = 'ready'") for batch in ready_batches { trigger_brainatlas_processing(batch) } }

Every 30 seconds, the watcher checks if any collecting batches have finished. When all fetch tasks for a batch are complete, it transitions the batch to ready status and triggers BrainAtlas processing. The frontend polls GET /batches/{id}/status every 2 seconds and sees the progress: "Fetching papers... 83/83 complete. Processing..." → "Generating summary..." → "Complete!"

Why Not Webhooks?

You might ask: why polling instead of webhooks? If the Fetcher could call back to the Orchestrator when a batch completes, wouldn't that be more efficient?

We considered it. Webhooks have problems:

  • Lost events: If the Orchestrator restarts during callback delivery, the event is lost.
  • Ordering: If callbacks arrive out of order, you need deduplication logic.
  • Authentication: The Fetcher needs credentials to call back, introducing circular dependencies.
  • Complexity: More code paths, more failure modes, harder debugging.

Polling is simpler. The database is the source of truth. The watcher is stateless—it can restart without losing progress. If it misses a cycle, it catches up on the next one. We tune the interval (30s) to balance latency vs. database load. For a research tool where summary generation takes 2-5 minutes total, 30-second granularity is acceptable.

End-to-End: From Query to Summary

Let's trace a complete request through the system. A researcher opens the CortexMap frontend, searches for "Hippocampus", and clicks "Generate Summary".

Phase 1: Query Generation (5 seconds)

The frontend calls POST /orch/regions/{hippocampus_uuid}/generate. The Orchestrator doesn't know what PubMed queries to use for the hippocampus, so it asks an LLM:

"Generate 5 diverse PubMed search queries to find research papers about the hippocampus. Include queries about anatomy, function, connectivity, disease, and electrophysiology. Return as structured JSON."

The LLM (GPT-4o-mini via OpenRouter) responds in 2-3 seconds with queries like:

  • "hippocampus[Title/Abstract] AND (anatomy OR structure OR cytoarchitecture)"
  • "hippocampal formation AND (memory OR learning OR consolidation)"
  • "CA1 OR CA3 OR dentate gyrus AND electrophysiology"
  • "hippocampus AND (alzheimer OR dementia OR neurodegeneration)"
  • "hippocampus AND (place cells OR grid cells OR spatial navigation)"

These queries are stored in the region_queries table for reuse. Next time someone generates a hippocampus summary, we skip this step.

Phase 2: Paper Fetching (60-180 seconds)

The Orchestrator creates a batch and submits all 5 queries to the Fetcher service. The Fetcher queries PubMed's E-utilities API, which returns PMC IDs. Typically:

  • Query 1 finds 20 papers (PubMed limits results to 20 per query)
  • Query 2 finds 20 papers (some overlap with Query 1)
  • Query 3 finds 15 papers
  • Query 4 finds 20 papers
  • Query 5 finds 18 papers

Total: 93 tasks created. After deduplication (same PMC ID from multiple queries), maybe 75 unique papers.

Workers start claiming tasks. Each worker:

  1. Calls PubMed's ESummary API for metadata (title, authors, abstract)
  2. Calls PubMed's EFetch API for full-text XML
  3. Uploads both to S3 at s3://cortexmap-papers/regions/{hippocampus_uuid}/{pmc_id}.xml
  4. Creates a papers table row linking PMC ID to S3 key
  5. Marks the fetch_task complete

With 5 workers running, 75 papers download in about 90 seconds (15 papers per worker, ~6 seconds per paper). The frontend shows real-time progress: "Fetching papers... 27/75 complete".

Phase 3: Completion Detection (up to 30 seconds)

The watcher's next cycle (within 30 seconds) detects all tasks are done. It transitions the batch to ready and invalidates Redis cache for the batch status key. The frontend's next poll sees status='ready' and displays "Processing...".

Phase 4: LLM Processing (120-240 seconds)

The watcher calls POST /brainatlas-be/api/process with the S3 keys of the 75 papers. BrainAtlas:

  1. Downloads papers from S3 (5 seconds for 75 XML files)
  2. Chunks text into 1000-char segments. 75 papers × ~6 chunks/paper = 450 chunks (2 seconds)
  3. Generates embeddings for all 450 chunks via OpenRouter API. With batching, this takes 30-45 seconds.
  4. Inserts embeddings into PostgreSQL. One transaction, 450 rows. (3 seconds)
  5. RAG loop: LLM makes 4 search calls over 3 rounds, reads 40 total chunks, generates summary. (60-90 seconds depending on API latency)
  6. Stores summary: Inserts into region_summary table, links to batch. (1 second)

Total: ~120 seconds. The watcher marks the batch completed and invalidates all relevant caches.

Phase 5: Frontend Display (instant)

The frontend's next poll sees status='Completed'. It fetches the summary via GET /orch/regions/{id}/summaries and renders:

"The hippocampus is a curved structure located in the medial temporal lobe [chunk:a3f7], consisting of several subregions including CA1, CA3, and the dentate gyrus [chunk:b8e2]. It plays a critical role in memory consolidation [chunk:c9f1] and spatial navigation [chunk:d4a6]..."

Each [chunk:id] renders as a small numbered bubble. Hovering shows a tooltip: "Source: PMC3847291, query: 'hippocampus AND spatial memory'". Clicking opens the paper.

Total end-to-end time: ~3-4 minutes from request to complete summary with citations.

Robustness in Practice

Distributed systems fail in creative ways. Here's how CortexMap handles real-world failure scenarios we encountered during development and testing.

Scenario 1: PubMed API Rate Limiting

What happened: We submitted 10 queries simultaneously. PubMed's E-utilities API returned HTTP 429 (Too Many Requests) for 7 of them.

System response: The Fetcher uses the backon crate for exponential backoff. Failed requests retry after 1s, then 2s, then 4s, up to 16s max. All 10 queries eventually succeeded within 30 seconds. No manual intervention.

What we learned: Retries need jitter. Without it, all 7 failed requests would retry at exactly the same time, causing another 429. We added ±25% random jitter to spread retry attempts.

Scenario 2: Malformed XML from PubMed Central

What happened: Paper PMC8473920 returned XML with an unclosed <title> tag. Our XML parser choked, worker crashed.

System response: The task's heartbeat timestamp stopped updating. After 60 seconds, the recovery job reset it to pending. Another worker claimed it, hit the same error, and the cycle repeated until we hit max retry limit (5 attempts). Task marked permanently failed with error message logged.

What we learned: Not all "failures" are transient. We added better XML validation and error categorization. Network errors get retried; parse errors get marked failed immediately. This reduced wasted retry attempts by 40%.

Scenario 3: BrainAtlas Service Crash Mid-Processing

What happened: We deployed a new BrainAtlas version with a memory leak. The service OOM-killed itself 90 seconds into processing a batch of 100 papers (after embedding 320 of 600 chunks).

System response: The watcher's next cycle (30s later) found the batch still in processing state. It checked the BrainAtlas health endpoint, got no response, and logged a warning. The batch remained in processing—we don't auto-reset processing batches because they might be legitimately slow.

What we learned: We needed better observability. We added a processing_started_at timestamp to batches. If a batch is in processing for more than 10 minutes, the watcher marks it failed and logs an alert. We also added memory limits to the Docker container.

Scenario 4: PostgreSQL Connection Pool Exhaustion

What happened: We spawned 20 Fetcher workers. Each worker holds a database connection. Default pool size: 10. Workers started failing with "connection pool timeout" errors.

System response: Workers logged errors but didn't crash. The heartbeat mechanism detected stalled tasks (workers couldn't update the database) and reset them after 60s. Tasks eventually completed when worker count dropped below pool size.

What we learned: We increased the pool size to 50 and added connection pool monitoring. We also added a max_workers config parameter that prevents allocating more workers than available connections.

Scenario 5: Deduplication Preventing Wasted Work

What happened: A researcher generated a hippocampus summary, then immediately requested another one (they didn't realize the first was processing).

System response: The second request triggered query generation, fetching, and got all the way to BrainAtlas processing. But before generating embeddings, BrainAtlas computed a SHA-256 hash of the downloaded paper content. It found an existing region_summary row with the same content_hash. It returned that summary's ID immediately without calling the LLM.

What we learned: Content-addressable caching is powerful. We extended this to short-circuit the entire pipeline: if a batch with identical fetch task PMC IDs exists and completed successfully, we skip fetching and return the existing summary. This handles the common case of "user clicks Generate twice in a row".

Design Decisions and Trade-offs

PostgreSQL as a Message Broker

Most distributed task queues use dedicated message brokers: RabbitMQ, Redis, AWS SQS. We use PostgreSQL. Why?

Benefits:

  • One less service: We already run PostgreSQL for data storage. Adding a message broker means another service to deploy, monitor, backup, and secure.
  • ACID transactions: Creating a batch, generating queries, and enqueueing fetch tasks happens in one atomic transaction. Either all succeed or all roll back. Can't achieve this with separate queue service.
  • SQL-based debugging: We can directly query SELECT * FROM fetch_tasks WHERE status='failed' to debug issues. No need to learn RabbitMQ management console or SQS CLI.
  • Proven scaling: PostgreSQL with SKIP LOCKED handles thousands of tasks/second. Our peak load is ~100 tasks/minute. We're not stressing it.

Trade-offs:

  • No pub/sub: PostgreSQL doesn't natively support fanout (one message to many consumers). We don't need it—each task has one consumer.
  • Polling overhead: Workers continuously poll for tasks. Adds database load. We mitigate with connection pooling and smart sleep intervals.
  • Not designed for this: Message brokers have features like message TTL, priority queues, and DLQs built-in. We implement these manually.

Would we switch to RabbitMQ at scale? Maybe. If we needed 10K tasks/second sustained, we'd revisit. For now, PostgreSQL is the right tool.

Self-Hosted LLMs vs. API Calls

The original project spec called for "self-hosted LLMs (BioGPT)". We use OpenRouter API instead. Why the change?

What we tried: We ran BioGPT (a GPT-2 derivative fine-tuned on PubMed) on university GPU hardware. Results were mixed:

  • Query generation: BioGPT generated syntactically correct PubMed queries but they were too narrow. Example: for "hippocampus", it generated "hippocampus[MeSH Terms]" which misses papers that use informal terminology.
  • Summarization: Summaries were factual but stilted. BioGPT struggled with complex sentence structures and transitional phrases.
  • Tool calling: BioGPT doesn't support function calling. We had to implement a custom parser for the RAG loop, which was brittle.

What we switched to: GPT-4o-mini via OpenRouter. Cost: ~$0.15 per summary (query generation + embedding + RAG synthesis). For 1,328 regions, total LLM cost is ~$200. This is acceptable for a research project.

The trade-off: We depend on an external API. If OpenRouter goes down or changes pricing, we're affected. For a production service with thousands of users, this might be unacceptable. For a research tool used by 10-20 concurrent users, the convenience and quality win.

We architected for swappability. The LLM interface is a trait. Switching to a self-hosted model requires implementing that trait, not rewriting the RAG loop.

Batch Processing vs. Streaming

Why do we fetch all papers before starting summarization? Why not stream: as each paper arrives, embed it and update the summary incrementally?

Batch advantages:

  • Deduplication: Content hashing only works if you have all papers upfront.
  • LLM coherence: Generating a summary from 50 papers in one RAG session produces better prose than merging 50 separately-generated snippets.
  • Citation consistency: Chunk IDs are stable. Streaming would require updating IDs as new chunks arrive.

Streaming advantages:

  • Lower latency: Users see partial results sooner.
  • Fault tolerance: If processing fails, you keep partial progress.

We chose batches because summary coherence is more important than latency for this use case. Researchers would rather wait 3 minutes for a good summary than get a mediocre one in 1 minute.

What We'd Do Differently

Observability from Day One

We added logging, but we didn't add metrics until we hit scaling issues. In hindsight, we should have instrumented with Prometheus and Grafana from the start. Key metrics we wish we'd tracked:

  • Task queue depth over time (detect backlog building)
  • Worker utilization (how often are workers idle?)
  • LLM API latency percentiles (P50, P95, P99)
  • Cache hit rates by endpoint
  • Database connection pool usage

Adding these post-hoc required sprinkling instrumentation throughout the codebase. Doing it upfront would have been easier.

Rate Limiting Earlier

We hit PubMed's rate limits multiple times during development before implementing proper throttling. We should have added rate limiting from the start:

  • Token bucket for PubMed API (10 requests/second max)
  • OpenRouter has per-model limits—track across parallel requests
  • Backpressure when workers exceed available database connections

Rate limiting is easier to build in than retrofit. We ended up with ad-hoc sleep(500ms) calls scattered through the code. A centralized rate limiter would be cleaner.

Better Error Categorization

Our initial retry logic was naive: retry everything up to 5 times. This wasted time on errors that would never succeed (malformed XML, HTTP 404). We should have categorized errors from the start:

  • Transient: Network timeout, rate limit, 503 server error → Retry
  • Permanent: 404 not found, parse error, invalid PMC ID → Fail immediately
  • Client error: Bad request, authentication failure → Alert and fail

This would have saved ~30% of retry attempts during development.

Structured Logging Sooner

We started with println! debugging. Switched to env_logger mid-project. Switched to tracing late in development. If we'd started with structured logging (JSON logs with correlation IDs), debugging distributed traces would've been easier.

Example: tracking a batch through the pipeline required grepping logs across three services. With correlation IDs, we could filter batch_id=abc123 and see the full journey.

Future Work

Interactive Exploration

Currently, summaries are static. Users can't ask follow-up questions like "Tell me more about place cells" or "How does the hippocampus interact with the prefrontal cortex?" We're exploring conversational RAG: users chat with the system, which retrieves relevant chunks on-demand and answers questions.

This requires session management and conversation history. The backend would maintain a per-user conversation state with retrieved chunks and LLM messages.

Federated Search

CortexMap currently covers the Allen Mouse Brain Atlas (1,328 regions). We'd like to expand to human, rat, and non-human primate atlases. Challenges:

  • Different region naming conventions (mouse "CA1" vs. human "hippocampus CA1 field")
  • Cross-species homology mappings
  • Scaling the database (10K+ regions × 50 papers/region = 500K papers)

This would enable comparative queries: "How does hippocampal neurogenesis differ between mice and humans?"

Real-Time Updates

Neuroscience research moves fast. Summaries become stale. We'd like to run periodic refreshes: every month, check PubMed for new papers, fetch them, and regenerate summaries. Challenges:

  • Detecting meaningful updates (new papers might not change the summary)
  • Showing diffs to users ("5 new papers added since last summary")
  • Cost management (1,328 regions × 12 updates/year = 16K summaries)

We're exploring incremental updates: embed new papers, search against them, and append to existing summaries instead of regenerating from scratch.

User Annotations

Researchers are domain experts. They might notice errors or want to add context. We'd like to support annotations: users can flag chunks as incorrect, add clarifying notes, or suggest additional citations.

This requires authentication, access control, and a moderation workflow. But it would make CortexMap a collaborative knowledge base instead of a read-only tool.

Conclusion

Building CortexMap taught us that AI-powered research tools require more than just an LLM API. The real engineering is in the infrastructure: distributed task queues that handle failures gracefully, vector databases that provide semantic search at scale, and orchestration layers that coordinate async workflows without blocking.

We didn't need Kubernetes or a service mesh. We needed clear service boundaries, PostgreSQL with smart features (SKIP LOCKED, pgvector), and careful attention to failure modes. The result is a system that processes thousands of papers, generates cited summaries in minutes, and runs reliably on modest infrastructure.

The code is ~10,000 lines of Rust across three services. The database schema is 10 tables. The deployment is a Docker Compose file. Complexity emerged from the problem domain (coordinating LLMs, managing async pipelines, tracking citations), not from over-engineering.

If you're building similar systems—ETL pipelines that feed LLMs, RAG applications that need citation tracking, distributed task queues for research workflows—we hope this writeup provides useful patterns. The full stack (Rust + PostgreSQL + pgvector + OpenRouter) proved remarkably capable.

CortexMap is open-source and deployed at Penn State for neuroscience research. We're excited to see what researchers discover using it.
